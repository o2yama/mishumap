# 決定記録

## 2026-07-09 プロジェクト始動時の決定

### ユーザー決定（AskUserQuestionで確認済み）

| 論点 | 決定 |
|---|---|
| 対象範囲 | 日本の全区分（星付き含む、約1,100軒）。ビブグルマンをデフォルト表示にする |
| 年代検索 | 最初から年次データを入れる |
| 地図/所要時間 | 無料構成: Leaflet + OpenStreetMap、直線距離ベースの徒歩◯分圏 |
| 公開 | 一般公開したい（ただし公開行為の実行はユーザー最終確認後） |

### データソースの決定

- **採用**: ngshiheng/michelin-my-maps（MIT）。最新CSV + 作者公開のDatasette履歴API
- **不採用**: guide.michelin.com の直接スクレイピング
  - 理由: 利用規約が自動収集を明示的に禁止。既に整備・公開されているデータセットがあるのに自前で規約違反リスクを負う理由がない
- 年次履歴はWayback復元由来で欠損あり（特に2020〜2023年）。v1はこのまま使い、UIに注記。
  将来の補完候補: 日本語まとめサイト（xn--e-3e2b.com の年度別ページ）からの手動転記

### 設計判断: アプリ構成（2案比較）

**A案: 静的SPA（Vite + JSON同梱、バックエンドなし）← 採用**
- 長所: ホスティング無料（GitHub Pages）・運用ゼロ・速い・データ更新はスクリプト再実行→コミットだけ
- 短所: データ更新に再デプロイが要る。全データをクライアントに送る（→日本限定1,100軒・gzip後数百KBなので問題なし）

**B案: バックエンドあり（Next.js + DB）**
- 長所: データ更新が動的、将来のユーザー投稿機能等に対応
- 短所: 運用コスト・インフラ管理が発生。v1の要件（読み取り専用の検索・表示）には過剰
- 却下理由: 要件が読み取り専用でデータが小さい。YAGNIに従いA案

### 設計判断: フロント実装（2案比較）

**A案: Vanilla TypeScript + Leaflet直接操作 ← 採用**
- 長所: Leafletは命令的APIでラッパー不要。依存が少なく軽い。フィルタUI程度なら状態管理は単純なstoreで足りる
- 短所: UIが大規模化すると手動DOM管理が辛くなる
- 相互批判への回答: v1のUIは「地図+フィルタパネル+リスト」で閉じており、React導入の複雑さ（react-leafletのバージョン互換問題含む）に見合わない

**B案: React + react-leaflet**
- 長所: 宣言的UI、将来の機能拡張に強い
- 却下理由: マーカー1,100個の更新はLeafletのlayer group差し替えのほうが素直で速い。規模に対して過剰

### 2026-07-09 レビュー（reviewer/Opus）を受けた修正

- **日本語検索の欠陥**: 店名・住所はデータ上ほぼ全てローマ字で「鮨」「渋谷」等がヒットしなかった。
  対応: 料理ジャンル58種の日本語エイリアス辞書（`src/cuisineAliases.ts`）＋カテゴリ・エリアの日本語ラベルを検索対象に追加。
  住所の日本語（渋谷→Shibuya）は未対応（v2候補: 地名辞書 or ローマ字変換）
- **hrefスキーム検証**: 外部データ由来URLに `javascript:` 等が混入した場合に備え、フロント（safeUrl）と生成側（http/https以外を除去）の二重防御を追加
- **キャッシュのデフォルト反転**: 旧仕様はキャッシュ優先で「更新したつもりが更新されない」footgunだった。デフォルト再取得・`--use-cache` はオプトインに変更
- **price="none" の素通し**: 表示時に空扱いへ修正
- **CARTOタイルの規約リスク（認識事項）**: 公開CDNタイルは非商用・attribution付きで利用可の扱いだが、トラフィック増時はレート制限があり得る。
  代替候補: OpenFreeMap（MapLibre化が必要）、タイルの自前ホスト。v1はCARTOで進める

### その他の技術決定

- 徒歩◯分の換算: **80m/分**（不動産の表示規約に準拠した慣習値）× 直線距離
- マーカー: canvasレンダラーの circleMarker（1,100点でもプラグインなしで軽い）
- 年フィルタの意味論: 「その年に受賞記録がある店」を表示。年次データに欠損があるため0件でも「掲載なし」とは限らない
