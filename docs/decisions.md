# 決定記録

## 2026-07-09 プロジェクト始動時の決定

### ユーザー決定（AskUserQuestionで確認済み）

| 論点 | 決定 |
|---|---|
| 対象範囲 | 日本の全区分（星付き含む、約1,100軒）。ビブグルマンをデフォルト表示にする |
| 年代検索 | 最初から年次データを入れる |
| 地図/所要時間 | 無料構成: Leaflet + OpenStreetMap、直線距離ベースの徒歩◯分圏 |
| 公開 | 一般公開したい（ただし公開行為の実行はユーザー最終確認後） |

### データソースの決定

- **採用**: ngshiheng/michelin-my-maps（MIT）。最新CSV + 作者公開のDatasette履歴API
- **不採用**: guide.michelin.com の直接スクレイピング
  - 理由: 利用規約が自動収集を明示的に禁止。既に整備・公開されているデータセットがあるのに自前で規約違反リスクを負う理由がない
- 年次履歴はWayback復元由来で欠損あり（特に2020〜2023年）。v1はこのまま使い、UIに注記。
  将来の補完候補: 日本語まとめサイト（xn--e-3e2b.com の年度別ページ）からの手動転記

### 設計判断: アプリ構成（2案比較）

**A案: 静的SPA（Vite + JSON同梱、バックエンドなし）← 採用**
- 長所: ホスティング無料（GitHub Pages）・運用ゼロ・速い・データ更新はスクリプト再実行→コミットだけ
- 短所: データ更新に再デプロイが要る。全データをクライアントに送る（→日本限定1,100軒・gzip後数百KBなので問題なし）

**B案: バックエンドあり（Next.js + DB）**
- 長所: データ更新が動的、将来のユーザー投稿機能等に対応
- 短所: 運用コスト・インフラ管理が発生。v1の要件（読み取り専用の検索・表示）には過剰
- 却下理由: 要件が読み取り専用でデータが小さい。YAGNIに従いA案

### 設計判断: フロント実装（2案比較）

**A案: Vanilla TypeScript + Leaflet直接操作 ← 採用**
- 長所: Leafletは命令的APIでラッパー不要。依存が少なく軽い。フィルタUI程度なら状態管理は単純なstoreで足りる
- 短所: UIが大規模化すると手動DOM管理が辛くなる
- 相互批判への回答: v1のUIは「地図+フィルタパネル+リスト」で閉じており、React導入の複雑さ（react-leafletのバージョン互換問題含む）に見合わない

**B案: React + react-leaflet**
- 長所: 宣言的UI、将来の機能拡張に強い
- 却下理由: マーカー1,100個の更新はLeafletのlayer group差し替えのほうが素直で速い。規模に対して過剰

### 2026-07-09 レビュー（reviewer/Opus）を受けた修正

### 2026-07-10 過去掲載店の表示（ユーザー要望）

### その他の技術決定

- 徒歩◯分の換算: **80m/分**（不動産の表示規約に準拠した慣習値）× 直線距離
- マーカー: canvasレンダラーの circleMarker（1,100点でもプラグインなしで軽い）
- 年フィルタの意味論: 「その年に受賞記録がある店」を表示。年次データに欠損があるため0件でも「掲載なし」とは限らない

### 2026-07-11 現在地取得のデフォルト化（ユーザー承認）

- ページを開いたら自動で現在地取得を試み、掲載エリア内なら**「現在地から徒歩15分圏内」をデフォルト表示**にする（従来はボタン押下時のみ）
- 自動取得の設計:
  - **エリア内判定は「最寄りの掲載店が10km以内」で近似**（`COVERAGE_RADIUS_M`）。掲載店は東京・大阪・京都・奈良にしかないため
    「4都市にいるか」の判定と実質同義。逆ジオコーディング等の外部API依存を増やさない
  - **位置ドットは許可されている限り常に表示**（ユーザー指示）。徒歩フィルタ・視点移動とは分離した
    - 圏外（例: 名古屋・福岡）ではデフォルトの全国表示のまま位置ドットだけ描画。距離順ソート・徒歩分表示もしない
    - `?area=`・`?awards=` 付きの流入（SEO一覧ページからの導線）ではフィルタ・視点は触らず、
      **許可済みのときだけ**（Permissions APIで確認）位置ドットを表示する。未許可なら許可ダイアログを出さない
    - 「現在地を解除」はフィルタ・並び替えの解除のみで、位置ドットは残す
  - 失敗（許可拒否・タイムアウト等）はユーザー操作起点でないため**エラー表示しない**。ボタンからの手動取得は従来通りエラー表示あり
- ハマりどころ（Leaflet）: **ズームアニメーション実行中の `fitBounds`/`setView` は無視される**。初期表示の
  `fitToResults()` のアニメーション中に自動取得が完了すると現在地へズームしなかったため、初期表示のfitを
  `animate: false` にして競合を排除した。あわせて `Circle.getBounds()`（ピクセル座標依存）をやめ
  `LatLng.toBounds()`（地理座標）でfit範囲を算出するようにした
- 検証: Playwright（位置情報を偽装）で 許可時（渋谷→15分圏8軒・円表示・ズーム15）/拒否時/SEOパラメータ流入/圏外（福岡→全国表示） の4シナリオを確認済み

### 2026-07-11 ポップアップの情報追加

- **公式ガイドの店舗写真の掲載は見送り**（ユーザー了承）。写真は紹介文以上に明確な著作物で、取得には公式サイトの
  スクレイピング（方針違反）が必要。一次ソースCSVに画像URLは存在しないことを確認済み。
  正規ルートはGoogle Places Photosだが課金＋プロキシ（バックエンド）が必要で静的サイト方針と衝突するため不採用
- **電話番号を追加**: データの96%（1,435/1,489軒）にE.164形式で存在する事実情報のため権利上の問題なし。
  `formatJaPhone()` で日本の慣習表記（03-XXXX-XXXX等）に整形し、`tel:` リンクで発信可能に。
  全1,435件がハイフン整形されることを確認済み（フォールバック0件）

### 2026-07-11 検索状態の保存・復元（ユーザー要望）

- フィルタ変更のたびに検索状態（区分・年・エリア・カテゴリ・検索語・過去掲載トグル・徒歩距離）を
  localStorage `bgm-last-search` に保存し、次回起動時に復元する（`src/persist.ts`）
- 設計判断:
  - **origin（現在地）は保存しない** — 古い位置の復元は無意味。起動時の自動取得に任せ、
    徒歩距離の選択値だけを引き継ぐ（現在地フィルタ使用中だった場合のみ保存）
  - **フィルタ指定付き流入（`?area=` 等）では保存状態を一切適用しない** — 保存済みの検索語まで復元すると
    SEOページが約束した一覧と違う結果になるため。パラメータ流入はクリーンな状態＋パラメータで表示
  - 保存データはバリデーションして現在のデータに存在する値だけ通す（古いエリア名・不正JSON対策）
- 検証: Playwrightで 変更→リロード復元 / SEO流入時の保存状態無視 / 既存4シナリオ回帰 を確認済み

### 2026-07-13 住所の日本語化（郵便番号データによる復元）

- 課題: 一次データ（michelin-my-maps）の住所・店名は英語/ローマ字のみで、日本語UIなのに
  「銀座」で検索して0件（実際は Ginza に61軒）という状態だった。日本人の検索行動と噛み合っていない
- **店名の日本語化は断念した**（現時点で手段がない）:
  - Google Places API: 利用規約で不可。「No use with a non-Google map」条項があり、本サイトは
    Leaflet + OSM タイルのため抵触する。加えて place_id 以外のキャッシュ・保存も禁止されている
  - OpenStreetMap (Overpass): 実測でカバー率が壊滅的。サンプル150軒に対し、名前一致で突合できたのは
    5軒(3.3%)、実際に日本語名が取れたのは3軒(2.0%)。最近傍マッチは同一ビルの別テナントを拾うため使えない
  - → 店名は原典どおりローマ字表記のまま。日本語化するなら人手しかない
- **住所の日本語化は自動化できた**（採用）:
  - 英語住所には必ず郵便番号が入っている（実測 1,106/1,106件）
  - 日本郵便のローマ字版郵便番号データ（KEN_ALL_ROME）で「郵便番号→都道府県・市区町村・町域」を
    漢字とローマ字の両方で引ける。同一郵便番号に複数町域がある場合は、英語住所側の町域ローマ字で
    照合して一意に決める（`scripts/address_ja.py`）
  - 郵便番号が町域を一意に決めている場合はローマ字照合を要求しない。長音の綴れ違い
    （Heinouchicho vs HEINOCHICHO）で落ちるため
  - 結果: **1,098/1,106軒（99.3%）を変換**。残り8軒は郵便番号自体が日本郵便の辞書にない（原典の誤り）
  - ライセンス: 日本郵便は郵便番号データについて著作権を主張していない（自由配布可）ため、
    生成した日本語住所をJSONに同梱して公開できる
- 反映先: 検索インデックス（`searchText`）・ポップアップ（日本語UIのみ）・SEO一覧ページの住所欄と
  JSON-LD（日本語ページのみ）。英語UI/英語ページは英語表記のまま
- 検証: 「銀座」61件・「渋谷」77件・「祇園」24件・「麻布」66件がヒット。ローマ字（Ginza）も61件で従来どおり

### 2026-07-13 初期ロードの分割（restaurants.json / details.json）

- 課題: 低速回線(Slow 4G)でピンが出揃うまで約6秒かかっていた。preloadで直列化は解消済みだが、
  律速が「順序」から「ペイロードの大きさ」に移っていた（JSON gzip 346KB のダウンロードそのもの）
- フィールド別に実測したところ `description` だけで全体の44.5%（raw 572KB）を占めていた
- **紹介文・公式URL・店サイト・電話の4項目を `details.json` に分離**し、ポップアップを開くまで読まない
  - 初期ロード: gzip **346KB → 73KB**。実測でJSON取得完了が 5.93秒 → 2.46秒（Slow 4G, キャッシュ無効）
  - 地図が描画された直後に裏で先読みする（`src/details.ts`）。通常ユーザーがピンを開く頃には届いている
  - 届く前に開かれた場合は、店名・住所・区分だけのポップアップを出し、**届いた時点で描き直す**
    （`renderPopupInto` + `popup.update()`）。検証: 到着前は紹介文/リンク/電話なし → 到着後に3つとも表示
  - 取得に失敗しても地図・検索は動く（ポップアップが簡素になるだけ）
- 区分名のコード化・座標の丸めも試したが gzip 後で2KBしか変わらないため採用しない（複雑さに見合わない）
- **静的ページ生成は両ファイルをマージする**（`gen-seo-pages.mjs`）。分割直後に店名リンク114本が
  消えるリグレッションを起こしたため、ビルド時は従来どおり全項目が揃っている状態にしてある

### 2026-07-13 掲載年の選択をスライダーからチップへ

- 課題: 年スライダーはトラックが4pxしかなく親指で掴みにくい。隣接年を踏みながら再描画が走るため、
  目的の年に着くまでに無駄なフィルタリングが繰り返される
- 掲載年を他の絞り込み（区分・エリア・カテゴリ）と同じチップUIに統一し、飛ばして選べるようにした
  - **最新年を先頭**に並べる（降順）。ほとんどのユーザーは最新版しか見ないので探させない
  - 単一選択。選択中のチップを再クリックしても選択は外れない（年が未選択という不正状態を作らない）
  - Lighthouse が指摘していた「`input#year-slider` にラベルがない」というA11y違反も同時に解消
- 年次データの欠損（2023年は26%等）は**注記1行のまま据え置く**とユーザーが判断（2026-07-13）。
  件数バーや復元率の明示、アーカイブ年のオプトイン化は見送り

### 2026-07-13 掲載年を複数選択できるようにする

- `FilterState.year: number` → `years: Set<number>`（区分チップと同じくOR条件のトグル）
- 意味論の設計判断:
  - **絞り込みはOR**（選択したいずれかの年に掲載されていればヒット）。区分チップと揃える
  - **表示する区分は「選択年のうち最も新しい掲載年」のもの**。2024=一つ星→2025=セレクテッドの店で
    両年を選ぶと「セレクテッド」を出す。同じ店の最新の格付けを見せるほうが誤解がない
  - includePast 併用時は「選択年のうち最新年より前の最後の掲載」まで遡る（従来の単一年と同じ考え方）
  - 全部外すと0件になる（区分チップを全部外した場合と同じ挙動なので許容）
- URL は `?year=2026,2025` のカンマ区切り。localStorage は `years` 配列で保存し、
  複数年対応前の単一 `year` も読めるようにしてある（既存ユーザーの保存状態を壊さない）
- 検証: 2026のみ362軒 / +2025で381軒 / +2024で386軒 と、データから事前計算した期待値に一致

### 2026-07-13 過去掲載トグルの廃止・現在地の自動取得の廃止・地図タップでシートを閉じる

いずれもユーザー指示（2026-07-13）。

**過去掲載トグル（`includePast`）を廃止**
- 年をチップで複数選択できるようになり、「その年に掲載されていた店を見る」は年チップで表現できる
- 付随して `EffectiveAward.isPast` も廃止（トグル専用の概念だった）。淡色表示・並び順・凡例の意味を
  **`inGuide=false`（最新版ガイドに載っていない＝閉店・掲載外れの可能性）基準に付け替えた**。
  過去年を選んだときに「今はもう無いかもしれない店」を見分ける手がかりは残す必要があるため
- 注意書きは年ヒント（最新年以外を選んだときに出る）へ統合。過去年を選んだときこそ必要な情報なので、
  表示条件として正しい

**現在地の自動取得を廃止**
- 起動時に位置情報の許可ダイアログを出さない。**徒歩距離のセレクトを選んだ時点**で取得する
  （「距離を選ぶ」こと自体が現在地から探したいという意思表示）
- 自動取得専用だった圏外判定（`COVERAGE_RADIUS_M`）と、前回セッションの徒歩距離の復元も撤去。
  現在地がない状態でセレクトに距離が入っていると「絞り込まれていないのに絞り込み済みに見える」ため、
  未取得時は必ず「距離指定なし」を表示する

**SPで地図をタップしたらシートを閉じる**
- `map.on("click")` に加えて `map.on("popupopen")` でも閉じる。ピンのタップは map の click に
  乗らない場合があるため

### 2026-07-13 価格帯フィルタ・ジャンル絞り込み・タップ領域

**価格帯フィルタ**
- 原典の価格表記は3形式が混在していた: `$$$`（1,102軒）/ `¥¥¥`（189軒）/ `4,000 - 6,000 JPY`（197軒）
- ミシュラン公式の価格帯（日本）に正規化して `priceLevel: 1〜4` を付与（`scripts/fetch_data.py`）

  - **金額レンジは中央値で帯に当てはめる**。記号の定義が「平均額」なので、レンジの中央が定義に最も近い

### 2026-07-13 ダブルタップ長押し＋上下ドラッグでズーム

- Googleマップと同じ片手ズーム操作を追加（`enableDoubleDragZoom` in `src/map.ts`）。Leafletに標準ハンドラがない
- **方向: 下へドラッグ = 拡大 / 上へ = 縮小**。これはデファクトで、以下の一次情報が一致している
  - Googleマップ公式ヘルプ（Android・Wear OS）: 「下にドラッグして、ズームインします。上にドラッグして、ズームアウトします」
  - Google Maps SDK for Android: "sliding the finger up to zoom out, or down to zoom in"
  - Mapbox GL JS の `tap_drag_zoom.ts`: `zoomDelta = dist / 128`（画面座標は下が正 → 下で拡大）。
    Google に合わせるため PR #9618 で意図的に反転された経緯がある
  - 注意: Leafletプラグイン Leaflet.DoubleTapDragZoom は**既定が逆**なので流用時は要注意
- 感度は 128px = 1ズームレベル（Mapbox GL JS と同値）
- 実装上の注意:
  - ドラッグ中はパン（`map.dragging`）とダブルクリックズームを止める。閾値6px未満で指を離した場合は
    通常のダブルクリックズームとして成立させたいので、その場合は無効化しない
  - `dblclick` は `pointerup` の直後に発火するため、`doubleClickZoom` の再有効化は1フレーム遅らせる
  - 拡大の基点は最初に触れた地点（`setZoomAround`）。Googleマップと同じく指の下が動かない
- 検証: 下256pxで+2レベル / 上256pxで-2レベル / 通常のダブルクリックで+1 / 通常のドラッグでパン、を確認
- **追記（同日）**: 実際に触ると「カクカクする」「基点が中心に流れる」という問題が出た。原因は
  Leafletの既定 `zoomSnap: 1`（整数ズームにしか止まれない）。小数ズームを指示しても丸められるため
  段階的に飛び、さらに `setZoomAround` が「丸める前のズーム」で中心を計算してから丸めるので基点がずれる。
  - `zoomSnap: 0` にして小数ズームを許可（ホイールズームも連続的になる副次効果あり）
  - 描画は `requestAnimationFrame` で1フレーム1回に間引く
  - 中心の計算は毎フレーム、**開始時の不変値（基点の緯度経度と画面座標）から** やり直す。
    `setZoomAround` は現在の地図状態を基準にするため、毎フレーム呼ぶと内部の丸めが累積し得る
  - 検証: ズーム15（1px≒3.9m）で ±1レベルの往復後、基点のずれ 0px
